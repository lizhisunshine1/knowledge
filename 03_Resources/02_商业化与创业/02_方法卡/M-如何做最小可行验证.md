# M-如何做最小可行验证

## 适用问题
如何用极低的心智摩擦、资金成本和时间成本，验证一个商业想法的真实性，避免“闭门造车半年，产品上线无人问津”的惨剧。

## 操作步骤
1. **定义最核心的单一假设（The Core Hypothesis）**：
   - 例如：“XX行业的运营人员愿意每月花99元购买一个自动化排版工具”。
2. **设计 MVP（Minimum Viable Product）**：
   - 绝不急着写代码或建网站。
   - 替代方案：做一个简单的宣传落地页（Landing Page）、发一个朋友圈调查、甚至直接做一个纯手动的微信社群服务。
3. **测试“利益交换”（Skin in the Game）**：
   - 检验 MVP 成功的唯一标准是**用户付出了实质性的代价**。
   - 接受标准（满足任意一项）：
     - 用户交了预付款/定金。
     - 用户留下了详细的个人联系方式并预约了演示（付出隐私和注意力）。
     - 用户花了一小时跟你做了深度需求访谈（付出时间成本）。
4. **快速迭代或放弃**：
   - 如果测试未达标，迅速寻找原因，修改假设重新测试，或果断放弃该方向。

## 注意事项
* **不要把“问卷调查”当成验证**：用户在不花钱的时候，口头支持率通常是 100%。

## 完整验证顺序（5 步）

不要停留在"分析"，要进入"验证"。

```
写清假设 → 验证需求 → 验证付费 → 验证交付 → 验证扩张
```

### 第 1 步：写清楚假设

把机会写成一句话：
> **为谁，在什么场景，解决什么问题，带来什么量化结果，靠什么赚钱。**

写不清 → 机会还没想透。

### 第 2 步：验证需求

找 5–10 个真实用户，问：
- 你现在怎么解决？
- 最大麻烦是什么？
- 为它付出过什么成本？
- 什么时候你愿意买？
- 预算从哪里来？

### 第 3 步：验证付费

不是问"你觉得有没有用"，而是：
- 你愿不愿意试点？
- 愿不愿意付一个小单？
- 愿不愿意提供数据和场景？
- 愿不愿意推动内部立项？

### 第 4 步：验证交付

做最小可行版本，看看能不能真的交付出结果。

### 第 5 步：验证扩张

一个点能成，再看到能否复制到更多客户、场景、组织。

## 可以复用到哪里
- 自媒体：录制抖音视频前，先在朋友圈发个选题大纲，看有多少人点赞评论。
- 解决方案：向客户推销大模型知识库方案前，先拿标准化架构图去探探客户的预算口风。
