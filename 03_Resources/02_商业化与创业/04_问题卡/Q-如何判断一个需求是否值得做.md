# Q-如何判断一个需求是否值得做

## 问题定义
当前有一个看似可行的需求，但无法确定它是否值得投入时间去验证或落地。

## 判断框架：需求 4 问

### 1. 频次够高吗？
用户多久遇到一次这个问题？
- 每天 → 高频，值得做
- 每周 → 中频，有条件可做
- 每月/每季 → 低频，除非单次价值极高

### 2. 痛点够痛吗？
不解决会怎样？
- 损失收入 / 增加成本 / 影响效率 / 带来风险 → 痛点强
- 只是"体验不好" / "不够方便" → 痛点弱

### 3. 有人愿意付费吗？
- 已经在为类似方案付费？→ 强信号
- 愿意为此改变工作流程？→ 强信号
- 说"挺好的"但不愿掏钱？→ 弱信号

### 4. 这是一个需求还是方案？
- "我想要一个 XX 工具" → 他说的是方案，不是需求
- "我需要解决 XX 问题，因为影响到 XX" → 这是需求

**需求永远问"为什么"，而不是"是什么"。**

## 快速判断表

| 高频 | 强痛 | 愿付费 | 结论 |
|:----:|:----:|:------:|------|
| ✅ | ✅ | ✅ | **值得做** |
| ✅ | ✅ | ❌ | 验证付费意愿 |
| ✅ | ❌ | ❌ | 不是真需求 |
| ❌ | ✅ | ✅ | 看单客价值 |
| ❌ | ❌ | ❌ | **跳过** |

## 可执行动作
- [ ] 找 5 个目标用户做需求访谈
- [ ] 问「你现在怎么解决这个问题？」
- [ ] 问「为解决这个问题你花了多少钱/多少时间？」
- [ ] 设计最小验证（一篇内容/一次咨询/一个表单）
